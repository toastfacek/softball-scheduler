# BGSL Design Context

## Product

**BGSL** (Beverly Girls Softball League) is a team coordination app for a youth softball team (12 players, 5 coaches, ~24 parent guardians). Used during a ~20-week spring season for game/practice scheduling, RSVPs, and inning-by-inning lineup planning.

## Users

- **Primary**: 5 coaches (Michelle, Mandy, Jamie, Jessica, Jesse — all parents of roster players). Mix of iPhone + Android + desktop use depending on the task.
- **Secondary**: ~24 guardians. They **never sign in**. They receive emails with tap-to-RSVP buttons (per-guardian signed links), reply inside the email, done.
- **Context**: coaches use the app from the field (phone), in the car, and at home (laptop the night before a game building lineups).

## Use cases

1. Coach creates a practice or game — broadcasts email to guardians with RSVP buttons
2. Guardian taps "Yes" in email → lands on `/rsvp/[token]` → confirmation
3. Coach checks who's in ahead of game day (Attendance screen — sorted by status, source-tagged)
4. Coach builds the lineup: drag batting order, tap cells to assign inning positions, eyeball the field visualizer
5. Coach sends a "practice canceled" update email
6. Admin adds new players / guardians / staff via Team page `+` (bottom sheet on mobile)

## Brand

- **Name**: Beverly Girls Softball League (team nickname: Beverly Blaze)
- **Colors**: navy + orange. Specifically:
  - `--navy-strong` (deep navy)
  - `--orange` / `--orange-strong` (warm, not neon)
  - Tinted paper + panel whites (never pure white)
  - Accent `--harbor` (blue-gray)
- **Fonts**: **Barlow Condensed** (display, 600/700) + **Sora** (body)
- **Personality**: Sporty but refined. Editorial sensibility — not a dashboard template. Feels like a real sports scorecard, not a SaaS app.

## Aesthetic rules

- **No AI slop tells**: no purple→blue gradients, no cyan-on-dark neon, no identical card-grids with icon+heading+blurb, no rounded card with colored stripe on one side, no glassmorphism-for-decoration, no glowing dark-mode default
- **Glass paper panels** with a soft warm-orange top-fade glow replace the old "hard orange stripe" stripe (kept in `.orange-bar-top` for continuity, but as a gradient)
- **Chip system** for status — not badges or pills
- **Barlow Condensed numbers** for dates, slot numbers, scoreboard-y moments
- **Cards are structural, not decorative** — list rows with thin dividers outperform a grid of identical cards
- **Asymmetry on desktop** — master/detail, lopsided grids; stacked on mobile
- **Motion is purposeful** — entrance animations only on the save-confirmation modal, no bouncy buttons

## Layout posture

- **Mobile-first**: slim 52px top bar, shared 4-tab bottom nav, single column ≤ 520px
- **Tablet** (640–1024px): widen content to 720-840px, keep bottom nav
- **Desktop** (≥ 1024px): switch to sidebar nav (top-left), multi-column content, Lineup editor becomes 3-pane (order / matrix / field viz)

## Stack constraints

- Next.js 16 App Router, Tailwind v4, server components by default
- No external UI libraries beyond lucide-react icons
- Design tokens live in `src/app/globals.css` — custom utilities + class components, not pure Tailwind
- Animations via CSS only (no framer-motion); exception: ConfirmModal's keyframes
